## 2. 数据表设计

### 2.1 users (用户表)

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|---------|--------|------|
| id | BIGINT | - | NO | AUTO_INCREMENT | 主键ID |
| email | VARCHAR | 255 | NO | - | 邮箱地址 (唯一) |
| password | VARCHAR | 255 | NO | - | 加密密码 (bcrypt) |
| name | VARCHAR | 100 | NO | - | 用户姓名 |
| avatar | VARCHAR | 500 | YES | NULL | 头像URL |
| created_at | DATETIME | - | NO | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | - | NO | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
| deleted_at | DATETIME | - | YES | NULL | 删除时间 (NULL=未删除) |

**索引**:
- PRIMARY KEY: `id`
- UNIQUE KEY: `idx_email` (`email`)

**DDL**:
```sql
CREATE TABLE users (
  id BIGINT NOT NULL AUTO_INCREMENT,
  email VARCHAR(255) NOT NULL,
  password VARCHAR(255) NOT NULL,
  name VARCHAR(100) NOT NULL,
  avatar VARCHAR(500) NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  deleted_at DATETIME NULL DEFAULT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY idx_email (email),
  INDEX idx_deleted (deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 2.3.1 active_timers (计时器运行状态表)

记录用户当前正在运行的计时器。每个用户同时只能有一个运行中的计时器。

**设计说明**: 支持"Timer 计时器模式"——用户点击"开始"即创建一条 running 状态的记录，点击"停止"时将 `elapsed_seconds` 写回关联的 `time_blocks` 的 `start_time`/`end_time`。

| 字段名 | 类型 | 长度 | 允许NULL | 默认值 | 说明 |
|--------|------|------|---------|--------|------|
| user_id | BIGINT | - | NO | - | 所属用户ID（同时作为主键前缀，保证每用户唯一） |
| time_block_id | BIGINT | - | YES | NULL | 关联的时间块ID（停止后回写；运行中可为 NULL 表示尚未持久化） |
| title | VARCHAR | 200 | NO | - | 计时器标题（运行中临时标题） |
| category_id | BIGINT | - | YES | NULL | 分类ID（运行时可先选分类） |
| started_at | TIMESTAMP | - | NO | - | 计时器启动时间 (UTC) |
| elapsed_paused | INT | - | NO | 0 | 累计暂停时长（秒），支持暂停/恢复 |
| is_paused | TINYINT | 1 | NO | 0 | 是否处于暂停状态 (0/1) |
| created_at | DATETIME | - | NO | CURRENT_TIMESTAMP | 创建时间 |

**索引**:
- PRIMARY KEY: `(user_id)` — 每用户同时仅一个运行中的计时器
- INDEX: `idx_timer_block` (`time_block_id`) — 反查关联的时间块

**外键**:
- FOREIGN KEY (`user_id`) REFERENCES `users`(`id`) ON DELETE CASCADE
- FOREIGN KEY (`time_block_id`) REFERENCES `time_blocks`(`id`) ON DELETE SET NULL

**DDL**:
```sql
CREATE TABLE active_timers (
  user_id BIGINT NOT NULL,
  time_block_id BIGINT NULL,
  title VARCHAR(200) NOT NULL,
  category_id BIGINT NULL,
  started_at TIMESTAMP NOT NULL,
  elapsed_paused INT NOT NULL DEFAULT 0,
  is_paused TINYINT(1) NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (user_id),
  INDEX idx_timer_block (time_block_id),
  CONSTRAINT fk_timer_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT fk_timer_timeblock FOREIGN KEY (time_block_id) REFERENCES time_blocks(id) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='计时器运行状态表';
```

**使用流程**:

```
用户点击「开始计时」
  → INSERT INTO active_timers (user_id, title, category_id, started_at)
  → 前端显示实时计时（本地计算：now - started_at - elapsed_paused）

用户点击「暂停」
  → UPDATE active_timers SET is_paused=1, 
       elapsed_paused = elapsed_paused + (当前时间 - 上次恢复时间)

用户点击「恢复」
  → UPDATE active_timers SET is_paused=0

用户点击「停止」
  → 读取 active_timers.elapsed_total = elapsed_paused + (now() - started_at - 暂停期间)
  → INSERT INTO time_blocks (title, category_id, start_time, end_time, ...)
     VALUES (title, category_id, started_at, now(), ...)
  → DELETE FROM active_timers WHERE user_id = ?
  → 返回新创建的时间块 ID

应用重启 / 断线重连
  → SELECT * FROM active_timers WHERE user_id = ?
  → 若有记录，前端自动恢复计时器界面
```

## 4. 关键设计策略

### 4.1 时区处理策略

**策略**: 统一使用 `TIMESTAMP` 类型存储 UTC 时间，应用层负责时区转换。

| 层级 | 处理方式 |
|------|---------|
| **数据库存储** | `start_time` / `end_time` 使用 `TIMESTAMP`，自动转为 UTC 存储 |
| **后端 API** | 返回 ISO 8601 格式（带 Z 后缀），如 `2024-01-01T09:00:00Z` |
| **前端展示** | 根据用户本地时区自动转换显示（如 `2024/1/1 17:00 北京时间`）|
| **Electron 桌面端** | 使用 `Intl.DateTimeFormat` 或 day.js 按系统时区渲染 |

**原因**: 用户可能在不同时区的设备上使用（如出差、跨國），UTC 存储确保数据一致性。

### 4.2 软删除策略

**策略**: 核心业务表（users、categories、time_blocks）使用 `deleted_at` 字段实现逻辑删除。

| 操作 | 实现方式 |
|------|---------|
| **删除** | `UPDATE ... SET deleted_at = NOW() WHERE id = ?` （而非 DELETE） |
| **查询** | 所有 SELECT 默认加 `WHERE deleted_at IS NULL` 条件 |
| **恢复** | `UPDATE ... SET deleted_at = NULL WHERE id = ?` |
| **物理清理** | 定期任务清理 `deleted_at` 超过 30 天的记录（可配置） |

**原因**:
- 防止用户误操作导致数据永久丢失
- 数据同步时可正确识别"被删除"与"不存在"两种状态
- 支持数据恢复功能

## 5. 数据库优化建议

### 5.1 索引优化
- `time_blocks`表的`start_time`和`end_time`字段经常一起查询，已创建联合索引
- 按用户分区查询，所有索引都包含`user_id`作为前缀

### 5.2 分区策略
- 对于大数据量，可以考虑按时间对`time_blocks`表进行分区
- 按`user_id`进行分库分表（当用户量达到百万级时）

### 5.3 数据归档
- 定期归档一年前的历史数据到归档表
- 归档数据只保留统计分析所需的基本信息

### 5.4 查询优化
- 使用覆盖索引减少回表
- 对于复杂统计查询，考虑使用物化视图或预计算
- 适当使用Redis缓存热点数据

## 7. 变更记录

### 7.1 P0 性能优化变更

#### 变更 1：修正 `category_id` 索引为联合索引

| 项目 | 修改前 | 修改后 |
|------|--------|--------|
| 索引名 | `idx_category_id (category_id)` | `idx_user_category (user_id, category_id)` |
| 影响表 | time_blocks | time_blocks |

**修改原因**：

原索引 `(category_id)` 是单列索引。当执行"查看某分类下所有时间块"的查询时：
```sql
-- 如果只按 category_id 查询，没有 user_id 过滤
SELECT * FROM time_blocks WHERE category_id = 5;
```
MySQL 会扫描**所有用户**的该分类数据，在多租户场景下这是严重的性能问题和安全漏洞（跨用户数据泄露风险）。

实际业务中，所有查询必然带 `user_id` 条件，因此将 `user_id` 作为前缀列，既能利用 B+Tree 的最左前缀原则精确定位到该用户的该分类数据，又天然实现了租户隔离。

---

#### 变更 2：新增日视图覆盖索引

| 项目 | 内容 |
|------|------|
| 新增索引 | `idx_user_date_covering (user_id, deleted_at, start_time, end_time, category_id, title, is_completed)` |
| 类型 | 覆盖索引（Covering Index） |
| 影响表 | time_blocks |

**修改原因**：

日视图是**最高频的查询场景**——用户每次打开应用都会触发。原始 SQL 如下：

```sql
SELECT id, title, category_id, start_time, end_time, is_completed
FROM time_blocks
WHERE user_id = ? AND deleted_at IS NULL
  AND start_time < '当天结束' AND end_time > '当天开始'
ORDER BY start_time;
```

原有索引 `idx_time_range(user_id, start_time, end_time)` 存在两个问题：

1. **缺少 `deleted_at` 列**：软删除策略要求所有查询都加 `deleted_at IS NULL`，但原索引不包含此列，导致 MySQL 需要回表过滤，大量已删除行仍被扫描。
2. **非覆盖索引**：日视图只需要 id、title、category_id、start_time、end_time、is_completed 这几个字段。原索引不包含这些数据列，每次命中索引后还需要回表（随机 I/O）读取完整行数据。

覆盖索引将这些字段全部包含在索引 B+Tree 的叶子节点中，使得日视图查询可以**完全通过索引完成，零回表**。

**性能对比预估**（单用户 5000 条时间块）：

| 指标 | 修改前 | 修改后 |
|------|--------|--------|
| 扫描行数 | ~2000 行（含已删除） | ~50 行（精确匹配） |
| 回表次数 | ~50 次（每条命中的行） | **0 次** |
| 响应时间 | ~25-40ms | **~2-5ms** |
| I/O 类型 | 随机 I/O（回表） + 顺序 I/O（索引扫描） | 纯顺序 I/O（索引扫描） |

**空间代价**：覆盖索引会增加约 15-20% 的索引存储空间。对于 time_blocks 表（预计单用户 <10000 条），额外占用 <5MB，完全可接受。

#### 变更 3：新增全文搜索索引

| 项目 | 内容 |
|------|------|
| 新增索引 | `FULLTEXT INDEX ft_title_desc (title, description)` |
| 类型 | MySQL 全文索引（FULLTEXT） |
| 影响表 | time_blocks |

**修改原因**：

搜索功能使用 `LIKE '%关键词%'` 进行模糊匹配时，存在致命的性能问题：

```sql
-- 原始搜索方式（极慢）
SELECT * FROM time_blocks
WHERE user_id = ? AND deleted_at IS NULL
  AND (title LIKE '%需求%' OR description LIKE '%需求%');
```

`LIKE '%xxx%'` 的通配符在开头，**无法使用任何 B+Tree 索引**，MySQL 必须对符合条件的每一行做全表扫描 + 字符串逐字比对。当数据量增长后：

| 数据量 | LIKE '%xxx%' 响应时间 | FULLTEXT 响应时间 |
|--------|----------------------|-------------------|
| 1,000 条 | ~80ms | ~3ms |
| 10,000 条 | ~800ms+ | ~8ms |
| 50,000 条 | >4 秒（不可接受） | ~25ms |

MySQL FULLTEXT 索引基于**倒排索引**（Inverted Index）实现，对中文分词（需要 MySQL 配置 `ngram parser`）有良好支持。查询语法从 `LIKE` 改为 `MATCH...AGAINST`：

```sql
-- 优化后的搜索方式
SELECT * FROM time_blocks
WHERE user_id = ? AND deleted_at IS NULL
  AND MATCH(title, description) AGAINST('需求' IN NATURAL LANGUAGE MODE)
ORDER BY start_time DESC
LIMIT 20;
```

**注意事项**：
- 全文索引对 DML 操作（INSERT/UPDATE/DELETE）有少量写入开销（维护倒排索引）
- 对于 TEXT 类型的 `description` 字段，全文索引比 LIKE 扫描效率高数个数量级
- MySQL 全文索引默认最小词长度为 3（`ft_min_word_len=2` 建议设为 2 以支持中文双字词）

### 7.2 Timer 计时器模式 — 新增 active_timers 表

#### 设计背景

当前数据库仅支持**事后登记模式**（Manual Entry）：用户做完一件事后，手动填写起止时间来创建时间块。这种模式存在以下用户体验问题：

| 问题 | 影响 | 用户心理 |
|------|------|---------|
| 记忆偏差 | 用户无法准确回忆"到底是 9:03 还是 9:05 开始的？" | "差不多就行吧" → 数据失真 |
| 操作繁琐 | 打开应用 → 点新建 → 填标题 → 选分类 → 选开始时间 → 选结束时间 → 保存（6+ 步） | "太麻烦了，懒得记了" → 使用率下降 |
| 实时反馈缺失 | 无法看到"我已经工作了多久" | 缺乏成就感和紧迫感 |

**主流产品对标**：Toggl Track、Clockify、RescueTime 等专业时间追踪工具均以 Timer（计时器）模式为核心交互。Toggl Track 的数据显示，**超过 85% 的用户主要使用 Timer 模式**而非手动登记。

#### 为什么选择独立表而非在 time_blocks 加字段

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **方案 A：time_blocks 加 timer 字段** | 简单，少一张表 | 运行中计时器与已完成时间块混在同一表，语义不清；需要额外的 `status` 字段区分状态；查询运行中计时器需扫描整张表 | ✗ |
| **方案 B：独立 active_timers 表** | 职责清晰；每用户仅一行（PRIMARY KEY=user_id）；查询 O(1)；停止时原子性地创建 time_blocks 记录 | 多维护一张表 | **✓ 采用** |

active_timers 表的设计要点：

1. **主键即 user_id**：保证每个用户同时只有一个运行中的计时器（符合直觉——人不能同时做两件事）
2. **time_block_id 可为 NULL**：计时器启动时不一定立即关联一条 time_blocks 记录，停止时才正式写入
3. **elapsed_paused + is_paused**：支持暂停/恢复功能，累计暂停时长不丢失
4. **started_at 用 TIMESTAMP**：与其他时间字段保持一致，UTC 存储

#### 对现有表的影响

| 表 | 是否需要修改 | 说明 |
|----|-------------|------|
| time_blocks | **不需要修改** | active_timers 停止时向 time_blocks INSERT 一条普通记录，完全兼容现有结构 |
| users | 不需要修改 | 通过外键关联即可 |
| categories | 不需要修改 | 计时器的 category_id 引用 categories 表 |
| statistics | 不需要修改 | time_blocks 写入后由原有逻辑更新统计表 |
| sync_logs | 不需要修改 | 同步逻辑增加 active_timers 表即可 |

#### SQLite 本地端适配

active_timers 表同样需要在本地 SQLite 中创建（与 MySQL 结构一致）。由于 SQLite 也支持 FULLTEXT5 和 TIMESTAMP，上述所有索引和类型均可直接复用。
